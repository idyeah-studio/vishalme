The Challenge

Merchants making million-dollar calls on inadequate tooling.

Luminate Insights is Walmart's merchant analytics platform — the tool category managers and buyers use to make inventory, pricing, and assortment decisions that affect hundreds of millions of dollars in product movement.

The platform needed high-density data visualization components that could handle radically different data schemas across product categories, be adopted by multiple engineering teams without forking, and maintain visual clarity at maximum data density. That's not a UI problem. It's an architecture problem that requires design and engineering fluency in equal measure.

Design Principles

Four decisions that shaped everything.

⚙️
Schema-agnostic by default
Components accept any data shape through a configuration layer. Product teams define behavior without touching internals.
🎯
Density without noise
Maximum data density with a strict visual hierarchy — the eye always has a clear entry point and a clear next action.
🔗
Table and chart as a system
Sorting, filtering, density modes, and color scales compose cleanly together rather than conflict at the boundaries.
⚡
Validated against real decisions
The real QA metric: a merchant scans a table and makes a reorder decision in under 30 seconds. Not a design preference.
My Approach

Designing and building in the same breath.

01

Configuration layer architecture

Designed the props API before the visual design — defining how product teams would configure column behavior, aggregation logic, and visualization type. This determined what the component could and couldn't do.

02

Visual hierarchy at enterprise density

Established a strict typographic and color hierarchy specifically for high-density data tables — ensuring critical values surface immediately even when surrounded by hundreds of data points.

03

React component build

Built the production React components directly — not just specced them. This closed the gap between design intent and implementation reality, and enabled the record release timeline.

04

Multi-team adoption without forking

Worked with each consuming team to validate the configuration layer covered their use cases. Where it didn't, extended the API rather than allowing forks — maintaining a single source of truth.
